# Sprint Chunk Summarizer

## Purpose

Read the per-project chunk files generated by the `sprint-chunk-extractor`
skill and produce a structured summary for each project. Each summary
captures completed work, pending work, blockers, next-week plans, and
risk signals. The summaries serve as input for `sprint-review-finalizer`.

## Invocation Model

This skill is invoked **once per chunk file** — one separate LLM call
per project. This keeps each invocation's context window small and
focused on a single project. For a CSV with 35 active projects, invoke
this skill 35 times (one per chunk).

All invocations are independent and can run **concurrently** since each
reads one chunk file and writes one summary file with no shared state.

## Input

- A single chunk file from `sprint_review_results/chunks/` (the user
  provides the file path, or the skill reads the next unprocessed chunk
  from the manifest).
- The manifest at `sprint_review_results/chunks/_manifest.md` (to know
  the full list of chunks and track progress).

## Procedure

1. **Read the manifest** at `sprint_review_results/chunks/_manifest.md`
   to get the list of all chunk files.
2. **Identify which chunks still need summaries** by checking which
   corresponding summary files already exist in
   `sprint_review_results/summaries/`.
3. **Pick the next unprocessed chunk file** (or use the file path
   provided by the user).
4. **Read the chunk file**.
5. **Analyse the data** across all assignees for that project.
6. **Produce a structured summary** (format below).
7. **Write the summary** to
   `sprint_review_results/summaries/<project_name_slug>.md`.
8. **Update the summaries manifest** at
   `sprint_review_results/summaries/_manifest.md` — append this
   project's row if the manifest exists, or create it if this is the
   first summary.
9. **Report** which chunk was processed and how many remain.

Do NOT cross-reference data between different project chunks — each
invocation is fully independent.

## Summary File Format

Write each summary to: `sprint_review_results/summaries/<project_name_slug>.md`

Each summary file must follow this structure:

```markdown
# <Project Name> — Sprint Summary

- **Week**: <date range>
- **Team**: <comma-separated list of assignees>
- **Total Planned Hours**: <sum of CURRENT WEEK planned hours across all assignees — do NOT include next-week plan hours>
- **Total Achieved Hours**: <sum of CURRENT WEEK achieved hours across all assignees>

## Project Classification

Classify this project as one of:
- **Delivery**: Feature development, implementation, deployment of new capabilities
- **Operational**: Infrastructure, maintenance, library upgrades, tooling, DevOps
- **Research**: Investigation, prototyping, design exploration, architectural review with no concrete deliverable this sprint

State the classification and a brief (1-sentence) justification based on the actions text.

## Completed Work

Bullet list of what was actually achieved this week. For each item:
- Name the person who did it.
- State the concrete outcome (e.g. "Refactored code and ran tests",
  "Completed unit tests, raising coverage >90%").
- Note if achieved hours differ significantly from planned hours
  (>20% over or under).

## Pending / Incomplete Work

Bullet list of planned work that was **not completed** or was
**partially completed**. For each item:
- Name the person.
- State what was planned vs. what actually happened.
- Note the gap (e.g. "Planned 24h, achieved 1h").

If all planned work was completed, write: "All planned work completed."

## Blockers & Dependencies

Bullet list of any blockers, delays, or external dependencies. Look for
these signals in the action text:
- "blocked by"
- "unable to complete"
- "delayed due to"
- "pending API specs"
- "dependency alignment"

For each item:
- Name the person.
- State what was blocked.
- State the dependency or reason (e.g. "blocked by DevOps",
  "pending API specs from upstream team").

If none found, write: "No blockers reported."

## Next Week Plan

Bullet list of what is planned for next week (from the next-week Plan
rows). For each item:
- Name the person.
- State the planned action and hours.

## Risk Signals

Identify and flag any of these risk patterns:
- **Hours variance**: Achieved hours significantly over or under planned
  (>20% delta).
- **Carried blockers**: Blockers from this week that are not resolved
  and may carry into next week.
- **Dependency chains**: Work that depends on output from another team
  or project.
- **Scope change**: Next-week plan differs significantly from what was
  originally planned continuation.
- **Under-staffing**: Only one person on a project that seems to need
  more capacity.

For each risk, state the signal, affected person(s), and potential impact.

If no risks identified, write: "No significant risks identified."
```

## Summaries Manifest

Write to: `sprint_review_results/summaries/_manifest.md`

```markdown
# Summaries Manifest

- **Source**: sprint_review_results/chunks/_manifest.md
- **Total projects summarised**: <count>

## Summaries

| # | Project | Classification | Team Size | Blockers? | Summary File |
|---|---------|---------------|-----------|-----------|--------------|
| 1 | Project 1 | Delivery | 3 | Yes | project_1.md |
| 2 | Project 4 | Operational | 2 | No | project_4.md |
| … | … | … | … | … | … |
```

## Rules

- Never invent data not present in the chunk files.
- Base the project classification solely on the action text in the chunk.
- If a cell shows "—", treat it as no action / no hours.
- Refer to people by their first name only.
- Create the `sprint_review_results/summaries/` directory if it does not
  exist.
- Overwrite any existing summary files from a previous run.
- Keep each summary factual and concise — no narrative filler.
- **Hours totals rule**: "Total Planned Hours" and "Total Achieved Hours"
  in the summary header must include **only the current week's** Plan and
  Achieved rows. Next-week Plan hours must NOT be added to these totals.
  Next-week data is used only in the "Next Week Plan" section.
- **Capacity comparison**: When comparing planned vs. achieved hours
  (for variance calculations, risk signals, etc.), always compare
  current-week planned hours against current-week achieved hours.
