# Sprint Review Finalizer

## Purpose

Read all per-project summary files produced by `sprint-chunk-summarizer`
and generate the final sprint review document. This is the last stage
of the three-skill pipeline.

## Input

- The summaries manifest at `sprint_review_results/summaries/_manifest.md`
- Individual summary files in `sprint_review_results/summaries/`

## Procedure

1. **Read the summaries manifest** to get the list of all summary files
   and their classifications (Delivery / Operational / Research).
2. **Read each summary file**.
3. **Produce the final report** with the sections defined below.
4. **Save** the report to `sprint_review_results/sprint_summary_<date>.md`
   where `<date>` is the week start date in YYYY-MM-DD format
   (e.g. `sprint_summary_2026-06-26.md`).

## Output Format

Produce the report in **GitHub-style markdown** with these sections in
order:

---

### Section 1 — 🗓️ Weekly Summary Header

State:
- The week date range (e.g. "Week of June 26 – July 2, 2026").
- Total number of active projects.
- Total number of people involved.
- A one-sentence executive summary of the sprint.

---

### Section 2 — Projects Table

A markdown table listing every active project:

| Project | Classification | Team Size | Status | Description |
|---------|---------------|-----------|--------|-------------|

Where:
- **Classification**: Delivery, Operational, or Research (from the
  per-project summary).
- **Team Size**: Number of assignees.
- **Status**: Derived from the summary — one of:
  - ✅ **On Track** — all planned work completed, no blockers.
  - ⚠️ **At Risk** — has risk signals or partial completion.
  - 🔴 **Blocked** — has active blockers preventing progress.
- **Description**: One-line description inferred from the actions text.

---

### Section 3 — 🌟 Key Project Highlight

**Scope**: Only **Delivery** projects. Exclude Operational and Research
projects from key project consideration — they are covered in their
own sections below.

**Auto-select** the single most important delivery project using this
priority order:

1. **Most progress made** — greatest delta between start-of-week state
   and end-of-week state (e.g. moved from design to deployment).
2. **Clearest measurable outcome** — numbers, percentages, milestones
   (e.g. "coverage >90%", "deployed to production").
3. **Closest to shipping** — deployment, release, go-live language in
   the actions.
4. **Highest risk** — active blockers or critical dependencies.
5. **Most hours invested** — highest total achieved hours.
6. **Most team members involved** — broadest team engagement.

Write the heading as:
`### 🌟 Key Project of the Week: <Project Name> – <Short Description>`

Then for **each delivery project** (key project first, then the rest),
write exactly **2 sentences**:
- **Sentence 1**: What was achieved this week vs. what was planned,
  noting anything incomplete.
- **Sentence 2**: What is planned for next week.

Format:
```markdown
**<Project Name>**: <Sentence 1>. <Sentence 2>.
```

The key project should be presented first with its highlighted heading,
followed by the remaining delivery projects as a bulleted list.

---

### Section 4 — ⚠️ Cross-Project Risks & Dependencies

Aggregate the "Blockers & Dependencies" and "Risk Signals" sections
from **all** per-project summaries (Delivery, Operational, and Research).

Group by dependency type:
- **DevOps Blockers**: Any work blocked by DevOps team.
- **Upstream API Dependencies**: Work waiting on API specs or external
  team deliverables.
- **Dependency Alignment**: Cross-team or cross-project alignment issues.
- **Capacity Concerns**: Under-staffing, significant hours variance,
  people spread too thin.
- **Carried Risks**: Blockers or issues from this week likely to persist
  into next week.

For each item, include:
- The person's first name.
- The affected project.
- What is blocked or at risk.
- The reason / dependency.

If no cross-project risks exist, write:
"No cross-project risks or dependencies identified."

---

### Section 5 — 🔬 Research Efforts Output

Summarise all projects classified as **Research** in the per-project
summaries.

For each research project:
- **What was investigated**: The research topic or area explored.
- **Key findings / progress**: Concrete outcomes — designs passed,
  prototypes built, architectural reviews completed.
- **Planned next steps**: What continues next week.
- **Delivery implications**: Flag if this research has implications for
  upcoming delivery projects (e.g. architectural decisions that affect
  feature work, design reviews that gate implementation).

If no research projects exist, write:
"No research-type projects this sprint."

---

### Section 6 — ⚙️ Operational Impact

Summarise all projects classified as **Operational** in the per-project
summaries (infrastructure, maintenance, library upgrades, tooling,
DevOps work).

For each operational project:
- **What was done**: Concrete actions taken (upgrades, deployments,
  refactoring, tooling integration).
- **Impact**: What this work achieved in terms of stability, performance,
  security, or unblocking other teams.
- **Remaining work**: What still needs to be done next week.
- **Delivery enablement**: Highlight any operational work that is
  actively blocking or enabling delivery projects.

If no operational projects exist, write:
"No operational-type projects this sprint."

---

### Section 7 — 📊 Capacity & Hours Overview

A summary table:

| Metric | Value |
|--------|-------|
| Total Planned Hours | <sum> |
| Total Achieved Hours | <sum> |
| Delta | <difference and percentage> |
| Total People | <count> |
| Avg Hours per Person (Planned) | <avg> |
| Avg Hours per Person (Achieved) | <avg> |

Then flag any noteworthy patterns:
- People significantly over capacity (achieved >> planned).
- People significantly under capacity (achieved << planned).
- Projects with hours variance >20%.

---

## Rules

- Never invent data not present in the summary files.
- Base all analysis on the per-project summaries — do not re-read the
  original CSV or chunk files.
- Refer to people by their first name only.
- Use emoji sparingly: 🗓️ for the header, 🌟 for the key project,
  ⚠️ for risks/blockers, 🔬 for research, ⚙️ for operations,
  📊 for capacity.
- Keep the tone concise and professional.
- Output only the markdown report — no preamble, no sign-off.
